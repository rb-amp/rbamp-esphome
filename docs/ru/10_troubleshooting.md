## 1. I²C / железо

### Probe failed at 0x50 — нет ответа

**Что видно в логе**:

```
[E][rbamp:xxx]: Probe failed at 0x50 — no response. Check wiring/address.
[E][component:xxx]: Component rbamp was marked as failed.
```

Компонент вызывает `mark_failed()` после этого и останавливает дальнейшие `update()`. Все привязанные сенсоры остаются в "Unknown" в Home Assistant.

**Что это значит**:

ESP32 отправил address-фазу на 0x50 (или другой адрес из `address:`) и не получил ACK. Модуль либо отсутствует, либо не запитан, либо на другом адресе.

**Как диагностировать**:

1. Проверьте, что в YAML-блоке `i2c:` стоит `scan: true`. Перезагрузите ESP32 и смотрите:

    ```
    [I][i2c.arduino:069]: Found i2c device at address 0x50
    ```

    Если скан нашёл устройство — ключ `address:` в вашем блоке `rbamp:` неправильный, обновите его.

2. Измерьте 3,3 В на пине VCC модуля мультиметром. Ноль или < 3,0 В — питание отсутствует или маргинальное.

3. Проверьте целостность SDA и SCL от GPIO ESP32 до коннектора модуля. Отсутствующий или перепутанный провод — самая частая причина на свежих сборках.

4. Проверьте pull-up резисторы на SDA и SCL (4,7 кОм к 3,3 В). Без подтяжек шина сидит в низком уровне, каждая address-фаза выглядит как NACK.

5. Если недавно выполняли смену адреса (ключ `new_address:`) — модуль может оказаться на новом адресе. Добавьте `scan: true` и найдите устройство; обновите `address:` соответственно и удалите `new_address:`.

**Фикс**: исправить разводку / питание, обновить `address:` если модуль переехал. Если модуль не отвечает после смены адреса — см. ["Адрес сменился, но модуль не отвечает"](#адрес-сменился-но-модуль-не-отвечает) ниже.

### RT block not ready — публикация пропущена

**Что видно**:

```
[D][rbamp:xxx]: RT block not ready (STATUS=0x00) — skipping live publish
```

**Что это значит**: статус-регистр сообщил что модуль не готов. Это происходит когда модуль ещё не закоммитил первый блок мгновенных значений после старта, в процессе flash-write или в момент мягкого ресета.

**Как диагностировать**:

1. Если запись появляется один раз при загрузке (в первые 30 с после питания) и потом исчезает — это нормально. Модулю нужен один цикл commit'а (~200 мс) чтобы выставить READY-бит после питания. Стартовый latch в `setup()` тоже даёт один пропущенный цикл.

2. Если сообщение льётся непрерывно после warm-up'а — модуль может застрять. Проверьте, нет ли повторных `ct_model:` / `ct_models:` записей (например ключ остался в YAML и пишется при каждой загрузке, давая постоянное окно flash-write).

3. Если сообщение появляется эпизодически (раз в несколько циклов) — нормальная вариативность связанная с NACK. Retry-цикл ловит большинство transient'ов; иногда чтение статуса всё же не проходит, тогда срабатывает эта DEBUG-запись.

**Фикс**:

- Warm-up пропуски при загрузке — ожидаемые; подождите два полных `update_interval` цикла.
- Если `ct_model:` / `ct_models:` стоит в YAML — оставьте: на каждом boot повторная запись идёт идемпотентно (модуль не записывает, если значение уже стоит в flash). Если warm-up пропуски досаждают — закомментируйте ключ после успешного первого пуска.
- Если модуль остаётся неготовым 60+ с — выполните power-cycle модуля (не ESP32) для чистого cold start.

### "rbamp took a long time for an operation (XXX ms)"

**Что видно**:

```
[W][component:522]: rbamp took a long time for an operation (406 ms), max is 30 ms
```

**Что это значит**: warning срабатывает когда `update()` любого компонента превышает дефолтный 30 мс budget кооперативного планировщика ESPHome. Для компонента rbamp это **ожидаемое и безвредное** поведение.

Модуль не имеет I²C auto-increment. Каждый float32 регистр требует 4 отдельных I²C-транзакций. Блок мгновенных значений на полной UI3 deployment читает до 9 float-регистров (U, I0–I2, P0–P2, PF0–PF2, Q0–Q2) — до 36 транзакций на 50 кГц ≈ 90 мс wire time плюс overhead. С retry (до 3 × 5 мс на байт при NACK-heavy цикле) общее время штатно 300–450 мс.

Latch-settle (`set_timeout("rbamp_period", 50, ...)`) **non-blocking** — `update()` отдаёт управление планировщику пока 50 мс проходят. Warning отражает wall-clock всего blocking-чтения мгновенных значений, не голодание WiFi или API.

**Действий от пользователя не требуется**. Warning косметический; сенсоры HA обновляются корректно.

**Если хотите снизить частоту warning'а**: объявите меньше сенсоров. Voltage-only или frequency-only deployment читает 1–2 регистра за цикл и укладывается в 30 мс. На полной UI3 (три канала + PF + Q) 400+ мс — особенность wire-протокола.

Cross-reference: [SPEC §6](https://rbamp.com/docs/modules-basic-standard-api-reference) (no auto-increment), [SPEC §B.5](https://rbamp.com/docs/modules-basic-standard-api-reference) (50 кГц).

### Не удалось прочитать period-snapshot

**Что видно**:

```
[W][rbamp:xxx]: Failed to read PERIOD_AVG_P[0]
[W][rbamp:xxx]: Failed to read PERIOD_AVG_P[1]
```

**Что это значит**: после latch-команды и 50 мс ожидания компонент не смог прочитать регистр средней мощности за период для канала N после трёх retry. Накопитель энергии для этого канала **не обновлён** в этом цикле.

Наиболее частые причины:

1. **Flash-write blackout**. Модуль выполняет запись CT-модели (триггер из `ct_model:` или `ct_models:` при загрузке) или сохранение нового адреса. Окно ~700 мс; если 50 мс latch-settle попадает в него — чтение таймаутит.
2. **Пропущенный latch при плотном опросе**. Под плотным back-to-back I²C-опросом возможен пропуск latch на стороне модуля. Снимок остаётся пустым, но valid-флаг будет 0 — компонент пропустит интегрирование с записью "Period snapshot not yet valid; keeping previous integration window". Warning срабатывает только когда valid-флаг выставлен, но чтение регистра всё равно упало.
3. **Transient NACK exhaustion**. Все три retry на хотя бы одном из четырёх байт float'а упали. На 50 кГц редко (< 1% циклов); на 100 кГц — чаще.

**Что происходит с энергией**: при warning'е `last_latch_ms_` **не** обновляется. В следующем успешном цикле `dt_ms` покрывает и пропущенное окно, и новое — энергия восстанавливается в предположении примерно стабильной нагрузки на промежутке. Это приемлемая малая неточность.

**Фикс**:

- Удалите ключи смены адреса (`new_address:`) из YAML после первой успешной загрузки.
- Если warning срабатывает чаще раза в час — проверьте скорость шины (`50kHz` рекомендуется) и pull-up резисторы (4,7 кОм).
- Если warning срабатывает на каждом цикле — проверьте, нет ли конфликта с другим устройством на шине.

### Адрес сменился, но модуль не отвечает

**Что видно**:

```
[E][rbamp:xxx]: Address change applied but slave not responding at 0x51 —
    check wiring and try power-cycling the rbAmp.
[E][component:xxx]: Component rbamp was marked as failed.
```

**Что это значит**: поток смены адреса прошёл через все шаги (запись нового адреса, сохранение во flash, мягкий ресет, переключение внутреннего I²C-адреса компонента), но модуль не ответил на новом адресе при первой проверке.

Возможные причины:

1. Адрес был записан и сохранён, но модуль зашёл в плохое состояние (повреждение параметров flash, glitch питания в окне 700 мс записи).
2. Модуль не был в factory-provisioning mode на момент попытки смены. Чтение статуса режима само могло отвалиться (NACK), и тогда компонент проводит запись спекулятивно с warning'ом в логе.
3. Проблема с разводкой, существовавшая до смены адреса (probe старого адреса успел только по совпадению на одном из трёх retry).

**Процедура восстановления**:

1. Модуль на неизвестном адресе. Используйте тестовый YAML с `i2c: scan: true` и цикл питания — найдите, на каком адресе модуль отвечает.
2. Если модуль найден на новом адресе (например 0x51) — обновите YAML на `address: 0x51`, удалите `new_address:`, перепрошейте. Следующий boot пройдёт чисто.
3. Если модуль не отвечает ни на одном адресе — запись во flash могла повредить параметры. Восстановление требует физического вмешательства: обратитесь к документации модуля или поставщику.

Cross-reference: [SPEC §10](https://rbamp.com/docs/modules-basic-standard-api-reference).

## 2. Качество данных

### Напряжение скачет — 228 В / 502 В / 1,96 В вперемешку

**Что видно**:

```
[D][sensor:xxx]: 'Mains Voltage': Sending state 502.730 V
[D][sensor:xxx]: 'Mains Voltage': Sending state 1.964 V
[D][sensor:xxx]: 'Mains Voltage': Sending state 228.365 V
[D][sensor:xxx]: 'Mains Voltage': Sending state 502.730 V
```

В Home Assistant график напряжения пилообразный между ~228 В, ~2 В и ~503 В. Ток одновременно может показывать 0 А или −2 А.

**Что это значит**: проявляются "ghost" значения из buffer leak драйвера ESP-IDF `i2c_master` под NACK-условиями на 100 кГц. Это **должно быть устранено** конфигурацией по умолчанию текущей версии компонента. Если симптом возник:

- В блоке `i2c:` стоит `frequency: 100kHz` — поменяйте на `50kHz`.
- Версия компонента старше 2026-05-24 — обновите.

**Фикс**:

```yaml
i2c:
  sda: GPIO21
  scl: GPIO22
  frequency: 50kHz   # обязательно для надёжной работы
  scan: true
```

Подробнее о механизме — [SPEC §B.5](https://rbamp.com/docs/modules-basic-standard-api-reference).

> **Совместимость с `framework: type:`** — NACK-проблема с ghost-значениями
> манифестируется только под драйвером ESP-IDF `i2c_master`. На
> `framework: type: arduino` (default) используется Arduino-фреймворковый
> `Wire`-стек, который под NACK ведёт себя иначе и не воспроизводит этот
> симптом. Это значит:
>
> - Если в YAML стоит `esp32: framework: type: arduino` — теоретически
>   можно держать `frequency: 100kHz` без видимых ghost-выбросов, но
>   per-byte retry-цикл компонента всё равно работает поверх Wire и
>   **периодические NACK при 100 кГц остаются** — просто без леденящих
>   ghost'ов. Для устойчивых графиков HA рекомендуем 50 кГц независимо
>   от framework.
> - Если в YAML стоит `esp32: framework: type: esp-idf` — `frequency: 50kHz`
>   **обязателен**. Это та конфигурация, на которой драйвер `i2c_master`
>   проявляет ghost-симптом самым ярко.
>
> Компонент сам не выбирает framework — он работает поверх обоих. Это
> исключительно решение вашего YAML.

### Ток постоянно 0 А под нагрузкой

**Что видно**:

```
[D][sensor:xxx]: 'Mains Current': Sending state 0.000 A
```

Мощность 0 Вт, энергия не растёт, но напряжение читается корректно (например 228 В).

**Что это значит**: одна из двух типичных причин:

**Причина 1 — `ct_model:` не выставлен или не соответствует физической клипсе**. На прошивке v1.2+ модуль использует заводские коэффициенты под выбранную в YAML модель. Если `ct_model:` не задан, или задан, скажем, `SCT_013_100` при реальной SCT-013-005 — диапазон измерения настолько шире физического сигнала, что разрешение AC-ADC даёт нулевые отсчёты.

**Причина 2 — клипса неправильно установлена** на проводнике (см. полярность в [03_sensor_selection.md](03_sensor_selection.md) и охват только одного проводника).

**Как диагностировать**:

1. Подтвердите, что клипса физически установлена на токонесущем проводнике и проводник действительно несёт ток (используйте клампметр как опорный, если есть).
2. Откройте YAML и проверьте, что `ct_model:` (или `ct_models:` для UI3) соответствует физической модели клипсы.
3. Если на прошивке v1.2+ выставлен `ct_model:`, но `sensor_class:` опущен или выставлен в значение, отличное от `SCT_013` — модуль откажет в записи модели. Лог при загрузке покажет warning с указанием на отсутствие предусловия.

**Фикс**:

```yaml
rbamp:
  id: meter1
  sensor_class: SCT_013       # дефолт; обязателен на v1.2+ перед ct_model
  ct_model: SCT_013_005       # соответствие физической клипсе
```

После загрузки модуль один раз запишет модель во flash (~700 мс) и подгрузит заводские коэффициенты. Следующая публикация мгновенных значений уже даст корректный ток.

**Точность измерения малых токов на UI2 / UI3** — паттерн dual-CT.
Если хотите одновременно высокое разрешение на малых нагрузках
(освещение, дежурный режим) и широкий диапазон для пиков (чайник,
стиральная машина) — задействуйте два канала тока на одной фазе
одновременно с разными клипсами:

```yaml
rbamp:
  id: home_meter
  sensor_class: SCT_013
  ct_models: [SCT_013_005, SCT_013_030, SCT_013_100]
  # CH0 — SCT-013-005 для разрешения idle-нагрузок (~5..50 Вт)
  # CH1 — SCT-013-030 для основной бытовой нагрузки (~50..7000 Вт)
  # CH2 — SCT-013-100 для пика (резерв EV-зарядки или генерации)
```

Все три клипсы охватывают **один и тот же** проводник фазы L.
SCT-013-005 даёт ~10× лучшее разрешение на малых токах относительно
SCT-013-030, но насыщается при ~5 А. SCT-013-030 покрывает «середину»
полезного диапазона; SCT-013-100 ловит пиковые броски без насыщения.
На стороне HA template-сенсор выбирает «активное» измерение по порогу:

```yaml
# configuration.yaml — выбор активного канала по диапазону
template:
  - sensor:
      - name: "Mains Current (active)"
        unit_of_measurement: "A"
        device_class: current
        state_class: measurement
        state: >
          {% set i0 = states('sensor.rbamp_ch0_current') | float(0) %}
          {% set i1 = states('sensor.rbamp_ch1_current') | float(0) %}
          {% set i2 = states('sensor.rbamp_ch2_current') | float(0) %}
          {% if i2 > 25 %}{{ i2 }}
          {% elif i1 > 4 %}{{ i1 }}
          {% else %}{{ i0 }}
          {% endif %}
```

Это полностью покрывает диапазон 5 мА..100 А с разрешением ~1 мА на
малых нагрузках. Trade-off: три CT-клипсы на одном проводнике
физически громоздки, и каждая занимает один канал модуля
(на UI3 это весь модуль). Для большинства deployment'ов одна клипса
с подходящим под нагрузку номиналом покрывает 99% сценариев — паттерн
dual/triple-CT нужен только когда требуется wide dynamic range.

### Period snapshot не валиден — энергия не растёт

**Что видно**:

```
[D][rbamp:xxx]: Period snapshot not yet valid; keeping previous integration window
```

Сенсор `energy` в Home Assistant показывает константу (или остаётся на 0 Wh после загрузки), хотя мощность читается корректно.

**Что это значит**: valid-флаг period-snapshot был 0 когда компонент его проверил через 50 мс после latch-команды. Модуль ещё не закоммитил валидный period-snapshot.

Нормальные причины:

1. **Первый latch после загрузки**. Стартовый latch в `setup()` всегда отбрасывает результат — накопитель модуля сбрасывается на первой latch-команде. Второй latch (через `update_interval` после загрузки) — первый, который интегрирует. Энергия начинает копиться после первого полного `update_interval` от boot.
2. **Cold start модуля**. Модулю нужен хотя бы один ~200 мс цикл commit'а для любых данных в period-накопителе. Если первый latch падает раньше — valid-флаг 0.
3. **Пропущенный latch при плотном опросе**. Под плотной I²C-нагрузкой возможен пропуск latch на стороне модуля. Снимок остаётся пустым; следующий latch должен пройти.

Если запись срабатывает на **каждом** цикле `update()` бесконечно (не только первые два) — модуль может быть в состоянии, когда period-накопитель никогда не коммитится. Power-cycle модуля сбрасывает это состояние.

## 3. Энергия и сохранение

### `energy_total` сбросилась в 0 после перезагрузки

**Что видно в Energy-дашборде HA**: резкое падение в 0 Wh, далее нормальное накопление от 0. Старая Wh-история сохранена в long-term statistics HA, но "текущее показание счётчика" выглядит как сброс.

**Что это значит**: восстановление из NVS не нашло данных или загрузило невалидные. Возможные причины:

1. **Bump версии NVS layout**. NVS-ключ выводится из ID компонента, XOR'енного с константой версии layout. Если layout структуры изменился (с апгрейдом компонента) — NVS-слот ключ меняется и старые данные перестают находиться. Восстановление тихо начинает с 0.
2. **Фрагментация или повреждение NVS**. Раздел NVS ESP32 использует flash wear-leveling. На очень старых deployment'ах или после множества прошивок фрагментация может вызвать сбой сохранения. В логе видно: `[W][rbamp:xxx]: Failed to save Wh totals to NVS — will retry next cycle`.
3. **Изменение разметки разделов**. Прошивка с новым `partitions.csv`, который меняет размер или положение раздела `nvs`, инвалидирует все сохранённые значения.
4. **Первая загрузка после factory reset или полного стирания flash**. Ожидаемо: NVS пустой, накопитель стартует с 0.

**Как диагностировать**:

Ищите при загрузке в логе:

```
[C][rbamp:xxx]: No saved energy state — starting from 0 Wh
```

Если присутствует — восстановление ничего не нашло. Также:

```
[C][rbamp:xxx]: Restored Wh from NVS: total[0.310, 0.000, 0.000] export[...]
```

Если восстановленное значение заметно ниже ожидаемого — последнее 5-минутное NVS-сохранение могло быть пропущено (худший случай потери: ≈ 5 Wh при средней 60 Вт).

**Фикс**:

- При апгрейде компонента, меняющем NVS layout, старые суммы не восстановятся автоматически. Long-term statistics HA сохранят историю даже если "текущее показание" обнулится.
- Чтобы сохранить Wh через принудительный wipe NVS — посмотрите текущие значения сенсоров в HA Developer Tools до wipe'а и запишите. Инжектировать стартовое значение в компонент способа нет.

---

### Энергия в HA не совпадает с эталонным счётчиком

**Что видно**: сенсор `energy` в HA читает заметно ниже (например 10–20%) эталонного smart-счётчика или счётчика энергокомпании за тот же период.

**Что это значит**: компонент использует ESP32-сторонний `millis()` интервал между latch-командами как интегрирующее время, а не таймер модуля. То есть дрейф часов модуля **не** влияет на точность накопленной энергии в компоненте.

Если энергия читается низкой — ищите среди:

1. **Загрузка по характеру**. Регистр period-средней мощности — действительно time-averaged мощность за закрытый период, а не мгновенное значение. Импульсная нагрузка (мотор, компрессор) с коротким on-time внутри 60-секундного периода усредняется ниже своей пиковой мощности.
2. **Положение клипсы и калибровка burden**. Заводская модель CT (`ct_model:`) не совпадает с реальной — диапазоны коэффициентов будут смещены. Проверьте, что `ct_model:` в YAML соответствует физической клипсе на проводнике.
3. **Тир BASIC прошивки**. На BASIC SKU per-window семплы P клампятся в 0 перед интегрированием. Если нагрузка имеет регенеративную компоненту (например мотор насоса с back-EMF) — BASIC под-считает.

## 4. Интеграция с Home Assistant

### HA показывает "Unavailable" для всех сенсоров rbamp

**Что видно в HA**: все сущности `sensor.rbamp_*` показывают "Unavailable".

**Причины и фиксы**:

1. **ESP32 офлайн**. Запустите `esphome logs --device <IP>` — если подключиться нельзя, ESP32 не в сети. Проверьте WiFi-креды, fallback AP если сконфигурирован.
2. **Соединение API потеряно**. Native API ESPHome (порт 6053) может блокироваться firewall'ом. Проверьте доступность ESP32 по IP:

    ```
    ping <esp32-ip>
    ```

    Если ping проходит, но HA всё ещё показывает unavailable — интеграцию ESPHome в HA нужно удалить и добавить заново (Settings → Integrations → ESPHome → Delete, затем re-discover).
3. **Несовпадение API encryption key**. Если сконфигурирован `api: encryption: key:` в YAML — интеграция HA должна знать тот же ключ. Несовпадение — HA считает устройство офлайн. Удалите и добавьте интеграцию заново после обновления ключа.
4. **mDNS не работает в вашей топологии сети**. Если ESP32 на другом VLAN от HA, multicast mDNS может не пересекать границу. Добавьте устройство по IP вручную в интеграции ESPHome.

---

### OTA-прошивка падает

**Что видно**:

```
ERROR Error resolving IP address: ...
# или
ERROR Upload Failed
```

**Частые причины**:

1. **IP-адрес сменился**. ESP32 получил новый DHCP-lease. Проверьте DHCP-таблицу роутера, найдите устройство по MAC и назначьте static IP, или используйте YAML-ключ `wifi: use_address:`.
2. **ESP32 в safe mode**. После трёх failed-boot подряд ESPHome заходит в safe mode (без кастомных компонентов, только WiFi + OTA). Safe mode OTA работает, но устройство крутит обрезанную прошивку. Маркер в `esphome logs`:

    ```
    [I][safe_mode:xxx]: Entering safe mode
    ```

    Форсируйте чистый boot: удерживайте BOOT-кнопку при включении, либо добавьте `safe_mode: num_attempts: 10` чтобы расширить порог.
3. **Порт 3232 (OTA) заблокирован**. OTA-протокол использует TCP 3232. Проверьте, что нет firewall-правила, блокирующего его между машиной с `esphome run` (или HA-хостом) и IP ESP32.
4. **Большая прошивка после добавления rbamp**. Первый OTA после добавления внешнего компонента может превысить размер OTA-раздела на дефолтной 4 МБ ESP32-разметке. Проверьте вывод `esphome compile` на warnings о размере binary'я и перейдите на разметку с большим OTA-разделом если нужно.

---

### OTA сразу после `new_address:` provisioning не загружается

**Симптом**: после успешной первой загрузки с ключом `new_address:`
(в логе виден `Address change SUCCESS, new address 0x51`) следующий OTA
падает с ошибкой compile или с runtime warning `address 0x50 NACK`.

**Что произошло**: модуль теперь живёт на новом адресе (0x51), но YAML
всё ещё указывает на старый (`address: 0x50`). При следующем `update()`
компонент пробует 0x50, ничего не получает, и логирует постоянные NACK.
OTA как таковой здесь не падает — это просто соседний симптом: вы
видите в `esphome logs` поток ошибок rbamp и решаете, что устройство
сломалось.

**Восстановление за 3 шага**:

1. Отредактируйте YAML:

    ```yaml
    rbamp:
      id: meter1
      address: 0x51        # новый адрес — выставили вместо 0x50
      # new_address: 0x51  # ← УДАЛИТЕ эту строку. Не комментируйте,
      #                       уберите целиком, иначе компонент попробует
      #                       снова провизионить и залогирует warning
      update_interval: 60s
      ct_model: SCT_013_030
    ```

2. Запустите `esphome run <yaml>` (через CLI) или Compile + Upload
   (через HA Add-on). Узел получит новый бинарь по WiFi OTA.

3. Откройте `esphome logs` и убедитесь, что в стартовом сообщении
   `dump_config` указан правильный адрес и `update()` идёт без NACK'ов.

**Правило** — `new_address:` это **одноразовая операция provisioning**.
После того как модуль переехал на новый адрес и сохранил его во flash,
ключ должен быть удалён из YAML. Иначе при каждой загрузке компонент
будет пытаться снова провизионить (на v1.2+ модуль игнорирует
повторную запись, но warning остаётся в логе и захламляет HA `notify`).

**Если модуль не отвечает ни на одном адресе после смены** — см.
["Адрес сменился, но модуль не отвечает"](#адрес-сменился-но-модуль-не-отвечает) выше.

## 5. Сборка и тулчейн

### Compile error: 'esphome/core/component.h' file not found

**Что видно в VS Code / clangd**: красные подчёркивания везде в `rbamp.h` и `rbamp.cpp`. Popup:

```
'esphome/core/component.h' file not found
```

**Что это значит**: clangd (C++ IntelliSense VS Code) не имеет доступа к include-путям ESPHome + PlatformIO. Build-система разрешает их корректно во время `esphome compile`; clangd — нет.

**Это не ошибка сборки**. Если `esphome compile example/ui1.yaml` проходит без ошибок — код корректен.

**Если хотите убрать красные подчёркивания**: создайте `.clangd` файл в директории компонента, указывающий на сгенерированные include-пути из вывода ESPHome build (`.esphome/build/<name>/src/`). Это косметика, опционально.

---

### PlatformIO падает с 0xC0000001D (EXCEPTION_ILLEGAL_INSTRUCTION)

**Что видно**:

```
Error: Failed to install Python dependencies (exit code: 3221225501)
```

(0xC0000001D = 3221225501 десятично)

**Что это значит**: PlatformIO 6.1.19+ использует `uv` Rust-бинарь для установки Python-зависимостей в свой внутренний `penv`. На CPU Intel Lunar Lake (Core Ultra 7 255H и подобных) `uv` падает с `EXCEPTION_ILLEGAL_INSTRUCTION` из-за несовместимости instruction set. Это влияет на шаг `penv_setup.py install` при свежей установке PlatformIO.

**Workaround** (предустановить все зависимости через pip, обойти `uv`):

```pwsh
# Шаг 1: установить Python-зависимости ESP32-платформы напрямую
& "$env:USERPROFILE\.platformio\penv\Scripts\python.exe" -m pip install `
    "littlefs-python>=0.16.0" "fatfs-ng>=0.1.14" "pyyaml>=6.0.2" `
    "rich-click>=1.8.6" "zopfli>=0.2.2" "intelhex>=2.3.0" "rich>=14.0.0" `
    "cryptography>=45.0.3" "certifi>=2025.8.3" "ecdsa>=0.19.1" `
    "bitstring>=4.3.1" "reedsolo>=1.5.3,<1.8" "esp-idf-size>=2.0.0" `
    "esp-coredump>=1.14.0" "pyelftools>=0.32"

# Шаг 2: bootstrap вложенного venv ESP-IDF (см. следующую запись)
& "$env:USERPROFILE\.platformio\penv\.espidf-5.5.4\Scripts\python.exe" `
    -m ensurepip --default-pip
& "$env:USERPROFILE\.platformio\penv\.espidf-5.5.4\Scripts\python.exe" `
    -m pip install -r `
    "$env:USERPROFILE\.platformio\packages\framework-espidf\tools\requirements\requirements.core.txt"
```

После установки этих пакетов `get_packages_to_install()` в `penv_setup.py` возвращает пустоту, и pip-install через `uv` не вызывается. Дальнейшие `esphome compile` работают штатно.

Workaround нужно применять повторно после свежей установки PlatformIO или очистки кеша `~/.platformio`.

---

### ESP-IDF venv: нет модуля `idf_component_manager`

**Что видно**:

```
ModuleNotFoundError: No module named 'idf_component_manager'
```

Появляется при первом `esphome compile` на машине, где PlatformIO скачал framework ESP-IDF, но не запустил полный bootstrap его requirements.

**Что это значит**: framework ESP-IDF от PlatformIO ставится во вложенный venv `~/.platformio/penv/.espidf-5.5.4/` (номер версии может отличаться). Этому venv нужен `requirements.core.txt` из пакета framework'а. Когда применяется `uv` workaround выше — шаг 2 bootstrap'ит pip в venv и ставит requirements.

Если ошибка возникла после применения `uv` workaround'а, но шаг 2 был пропущен:

```pwsh
# Найти venv IDF (номер версии может отличаться):
ls "$env:USERPROFILE\.platformio\penv\" | Where-Object { $_.Name -like ".espidf-*" }

# Bootstrap pip в venv IDF:
& "$env:USERPROFILE\.platformio\penv\.espidf-5.5.4\Scripts\python.exe" `
    -m ensurepip --default-pip

# Установить core requirements IDF:
& "$env:USERPROFILE\.platformio\penv\.espidf-5.5.4\Scripts\python.exe" `
    -m pip install -r `
    "$env:USERPROFILE\.platformio\packages\framework-espidf\tools\requirements\requirements.core.txt"
```

Подставьте свой номер версии IDF вместо `5.5.4` (посмотрите в `~/.platformio/penv/`).

### `esphome compile` падает: `get_object_id_hash` не найден

**Что видно**:

```
error: 'class esphome::rbamp::RbAmpComponent' has no member named 'get_object_id_hash'
```

**Что это значит**: старая версия `rbamp.cpp` вызывала `this->get_object_id_hash()` для seed'а NVS-слота. Этот метод убран из `PollingComponent` в ESPHome 2025.x.

**Фикс**: обновите компонент до текущей версии. Современный код использует hash-формирование совместимое с ESPHome 2025.x и новее.

---

### ESPHome слишком старый — `set_i2c_address` не найден

**Что видно**:

```
error: 'class esphome::i2c::I2CDevice' has no member named 'set_i2c_address'
```

**Что это значит**: поток смены адреса `new_address:` вызывает `i2c::I2CDevice::set_i2c_address()`, добавленный в ESPHome 2023.6.

**Фикс**: обновите до ESPHome 2023.6 или новее. HA ESPHome Add-on отслеживает текущий релиз автоматически — рекомендованный путь установки.

```sh
# Если локальный Python venv:
pip install --upgrade esphome
```
