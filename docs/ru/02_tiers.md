# 02 · Тиры модулей

rbAmp поставляется в трёх продуктовых линейках: **BASIC**, **STANDARD** и **PRO**. Каждый тир — это законченная комбинация аппаратной ревизии и сборки прошивки, не просто флаг в прошивке. Переход между тирами требует физической замены модуля, а не апгрейда.

ESPHome-компонент работает со всеми тремя тирами одинаково: YAML-схема общая, отличается только то, какие данные модуль действительно отдаёт и какие YAML-ключи имеют смысл на каждом тире.

## Тиры

### BASIC — базовый

Аналоговый front-end оптимизирован по стоимости, подходит для типовых бытовых нагрузок. Прошивка ведёт **однонаправленный учёт потребления**: средняя активная мощность за период не может быть отрицательной — реверсные участки внутри окна нулятся перед интегрированием. По поведению похоже на классический механический счётчик с диском.

- Сенсор `energy_exported` существует в YAML-схеме (для совместимости конфигов), но на BASIC всегда публикует `0`.
- Типичные применения: квартиры без собственной генерации, sub-метеринг конкретных потребителей (бойлеры, моторы, бытовая техника, климатика).

### STANDARD — двунаправленный

Расширенная аналоговая стойка для точного двунаправленного измерения. Прошивка ведёт **два раздельных накопителя** за период: один для потребления (положительные участки), второй для экспорта (отрицательные). Сенсор `energy_exported` на этой линейке несёт реальный сигнал.

- Ключ `bidirectional: true` на блоке `rbamp:` включает чтение регистра экспортной энергии.
- NVS-сохранение работает для обоих накопителей одинаково.
- Типичные применения: дома с солнечной/ветровой генерацией, аккумуляторные накопители, V2G-зарядки EV, любая установка где импорт и экспорт нужно учитывать раздельно.

### PRO — премиум

Малошумный аналоговый front-end, более жёсткие допуски на компоненты. На PRO дополнительно доступны:

- Расширенная диагностика (peak demand windows, индикаторы гармоник, мониторинг дрейфа) — будет интегрирована в YAML-схему в одной из следующих ревизий компонента.
- SKU с большим числом каналов (UI5 / UI7, I6 / I8).

Типичные применения: sub-метеринг коммерческих арендаторов, billing-grade установки, инструментальные лаборатории, энергоёмкие промышленные нагрузки, service bureau с требованием audit-quality данных.

| Возможность | BASIC | STANDARD | PRO |
|---|:---:|:---:|:---:|
| Среднеквадратичное напряжение | да | да | да |
| Среднеквадратичный ток (1..3 канала) | да | да | да |
| Активная мощность (signed float) | да | да | да |
| Коэффициент мощности | да | да | да |
| Реактивная мощность | да | да | да |
| Полная мощность | да | да | да |
| Частота сети | да | да | да |
| Средняя мощность за период (импорт) | да | да | да |
| Средняя мощность за период (экспорт) | всегда 0 | да | да |
| Сенсор `energy` (импорт Wh, NVS) | да | да | да |
| Сенсор `energy_exported` (NVS) | всегда 0 | да | да |
| Расширенные диагностические регистры | нет | нет | да |
| SKU с большим числом каналов (UI5 / UI7 / I6 / I8) | нет | нет | да |

> Базовый блок регистров одинаков на всех тирах. STANDARD- и PRO-специфичные регистры лежат в отдельном диапазоне выше базового. Компонент читает только то, под что объявлены сенсоры — необъявленный регистр не запрашивается никогда.

## YAML-ключи по тирам

### `bidirectional:` (по умолчанию `false`)

Включает ветку чтения регистра экспортной энергии в каждом цикле `update()`.

```yaml
rbamp:
  id: solar_meter
  address: 0x51
  bidirectional: true   # имеет смысл на STANDARD / PRO; на BASIC — no-op
```

На BASIC ключ принимается схемой без ошибки, но `energy_exported` всегда даёт 0. Способа определить тир в рантайме у компонента нет (тирового регистра в текущей прошивке не предусмотрено) — пользователь сам выставляет ключ под свой SKU.

> Ключ зарезервирован под предстоящие ревизии прошивки. На текущей сборке компонент уже принимает его в схеме и резервирует слоты `energy_export_wh[3]` в NVS, но реальное чтение экспортного регистра подключится в одной из следующих ревизий компонента вместе с финальной раскладкой регистров.

### `sensor_class:` — фиксация семейства датчика

```yaml
rbamp:
  id: meter1
  sensor_class: SCT_013   # по умолчанию SCT_013
```

Принимается на всех тирах. На прошивке v1.2+ значение записывается во flash и становится **предусловием** для записи CT-модели: модуль откажет в записи модели если класс не выставлен. На более ранних прошивках значение принимается схемой и применится автоматически при апгрейде прошивки — без необходимости менять YAML.

Значения enum: `SCT_013` (текущее семейство), `WIRED_CT`, `BUILTIN_CT` (зарезервированы под будущие SKU).

### `ct_model:` / `ct_models:` — модель CT-клипсы

Один глобальный профиль для всех каналов:

```yaml
rbamp:
  id: meter1
  ct_model: SCT_013_030   # все каналы — SCT-013-030
```

Или индивидуальный профиль на канал (для UI3 со смешанными клипсами):

```yaml
rbamp:
  id: meter_ui3
  ct_models: [SCT_013_005, SCT_013_030, SCT_013_100]
```

Принимаются на всех тирах, взаимоисключающиеся. На прошивке v1.2+ модуль автоматически подгружает заводские коэффициенты под выбранную модель — никаких дополнительных calibration-шагов не требуется. На ранних прошивках значение сохраняется как пометка модели; полноценный auto-load коэффициентов придёт с апгрейдом прошивки.

Подробнее о выборе модели — в [03_sensor_selection.md](03_sensor_selection.md).

### `new_address:` — однократная провизия I²C-адреса

```yaml
rbamp:
  id: meter1
  address: 0x50
  new_address: 0x51   # одноразовый; удалить после успешной загрузки
```

Принимается на всех тирах. Требует **factory-provisioning mode** на стороне модуля — это режим, в который модуль переводится для одноразовых операций (смена адреса, factory reset). Стандартные production-модули поставляются с отключённым provisioning-режимом — обратитесь к документации модуля или поставщику за процедурой включения.

Поток выполнения за один boot: компонент опрашивает текущий адрес, пишет новый адрес и сохраняет во flash, сбрасывает модуль, проверяет ответ на новом адресе. После успеха обновите YAML на `address: 0x51` и удалите `new_address:`. Если компонент обнаружит, что старый адрес молчит а новый отвечает, он адаптируется автоматически с warning'ом в лог (на случай если предыдущая загрузка уже применила смену).

### `broadcast_latch:` — синхронный latch для multi-module шины

```yaml
rbamp:
  - id: meter1
    broadcast_latch: true   # no-op на v1 прошивке; safe для forward-compat
```

При `true` компонент отправляет latch-команду на широковещательный I²C-адрес — за одну транзакцию синхронизирует period-snapshot на всех модулях шины. **На v1 прошивке general-call отключён** — запись на широковещательный адрес тихо игнорируется. Компонент видит версию прошивки и логирует `ESP_LOGW` warning, затем откатывается на последовательный latch для каждого модуля по отдельности.

Skew между модулями на 50 кГц шине — порядка ~540 мкс × N, что меньше 0,003% от 60-секундного периода. На дневных суммах HA эта неровность незаметна.

Оставляйте ключ в YAML при апгрейде с v1 на v1.1 прошивку — warning исчезнет, компонент сам перейдёт на настоящий broadcast.

### `topology:` — подсказка о топологии

```yaml
rbamp:
  id: meter1
  topology: SINGLE      # SINGLE / SPLIT_PHASE / THREE_PHASE
```

Принимается на всех тирах. На текущей прошивке компонент сам выводит число каналов из объявленных слотов сенсоров (если объявлен `current_1` — каналов как минимум 2). Ключ — косметический, идёт в строку `dump_config` лога. Станет авторитативным когда модуль начнёт публиковать топологию через свой регистр в одной из следующих ревизий прошивки.

### `drdy_pin:` — опциональное прерывание готовности

```yaml
rbamp:
  id: meter1
  drdy_pin: GPIO15
```

Принимается на всех тирах. Подключается к необязательному open-drain выходу DRDY модуля. Когда объявлен — компонент использует пин как подсказку, что мгновенные регистры свежие до их чтения. При типовом `update_interval: 60s` ключ можно опустить: компонент пропускает мгновенные чтения если status-регистр не готов, независимо от DRDY.

Требует pull-up 10 кОм на GPIO мастера к 3,3 В (выход open-drain, внутренней подтяжки на стороне модуля нет). См. [04_hardware.md](04_hardware.md).
