## Политика deprecation

Что компонент обещает не ломать без предупреждения, а что — может:

| Surface | Стабильность | Что обещаем |
|---|---|---|
| YAML-ключи блока `rbamp:` | **Стабильно** | Удаление или переименование возможно только в major-bump (v1.0.0). Через 2 минорных релиза до удаления — deprecation warning в логе `dump_config`. |
| YAML-ключи `sensor.platform: rbamp` (slots) | **Стабильно** | Те же правила. Имена `voltage`/`current`/`power`/`energy`/`frequency`/`power_factor` зафиксированы как контракт с HA Energy-дашбордом. |
| Состав HA-сущностей и их `device_class` / `state_class` | **Стабильно** | Изменение не сделаем без объявления; затрагивает entity_id в HA. |
| Внутренние lambda-доступные методы `RbAmpComponent::*` | **Не-стабильно** | Могут меняться в минорах. Если вы пишете lambda-actions против внутренних API — закрепите версию компонента (`refresh: <git-sha>` в `external_components:`). |
| Формат NVS-структуры | **Версионируется** | Изменения сопровождаются bump'ом `RBAMP_PREF_VERSION` — старые данные тихо отбрасываются, не повреждаются. Это значит ровно один эпизод 5-минутной потери энергии при апгрейде, не больше. |
| Имена и поведение `i2c:` блока | **Внешнее** | Это ESPHome core, не наше — следуйте политике upstream. |
| Поверхность wire-протокола (регистры модуля) | **SemVer прошивки** | Smart-rules SPEC: minor добавляет регистры; major может перенумеровать. Компонент детектит версию прошивки и адаптирует поведение, см. таблицу совместимости в [README §Совместимость](../README.md). |

**Что значит «deprecation warning»**: за два минора до удаления ключ
начинает логировать предупреждение при загрузке. Это сигнал поменять
YAML до следующего minor-релиза. Никаких silent breakages в стабильных
surfaces — если что-то всё-таки сломается без предупреждения, это баг,
заводите issue.

**Pre-1.0 caveat**: текущие релизы 0.x — это alpha/beta-серия. Мы
стремимся следовать политике выше уже сейчас, но оставляем себе
небольшое пространство для критичных breaking changes в минорах
до v1.0.0 — каждое такое изменение явно отметим в этом changelog с
тегом `[BREAKING]` и инструкциями по миграции.

## v0.4.0 — 2026-05-28, паритет с протоколом v1.2

Компонент достигает паритета с протоколом прошивки v1.2 одновременно с сестринскими библиотеками Arduino, ESP-IDF и Python v1.1.0.

### Новые YAML-ключи (v0.4.0)

- **`sensor_class:`** — `SCT_013` | `WIRED_CT` | `BUILTIN_CT`, по умолчанию `SCT_013`. Фиксирует семейство датчика тока на стороне модуля. На прошивке v1.2+ становится предусловием записи модели CT-клипсы. На ранних прошивках принимается схемой и применится автоматически при апгрейде.
- **`ct_models:`** — список из **1–3** enum-значений `SCT_013_*` для UI2/UI3 со смешанными моделями клипс на разных каналах (схема: `cv.Length(min=1, max=3)`). Взаимно исключающий с глобальным `ct_model:`.

### Изменённое поведение

- **`ct_model:` теперь подгружает заводские коэффициенты** под выбранную модель на прошивке v1.2+ — записывает и noise floor, и gain автоматически. На более ранних прошивках работает как в v0.3 (metadata-only).
- **Per-channel конфигурация для UI3** через `ct_models: [SCT_013_005, SCT_013_030, SCT_013_100]` — компонент сам ставит свою модель на каждый канал при загрузке. Общее время setup'а для трёх каналов — ~2-3 с (per-channel flash-write).

### Кросс-платформенный паритет

Полный паритет с Arduino + ESP-IDF + Python v1.1.0:

| Surface | Arduino | ESP-IDF | Python | ESPHome v0.4.0 |
|---|:---:|:---:|:---:|:---:|
| Класс датчика (`sensor_class:`) | ✓ | ✓ | ✓ | ✓ |
| Per-channel CT-модель | ✓ | ✓ | ✓ | ✓ |
| Предусловие класса для записи модели | ✓ | ✓ | ✓ | ✓ |

### Breaking changes

Нет. Все изменения additive — старые YAML без новых ключей продолжают работать как раньше.

### Миграция с v0.3

Для типовых deployment'ов действий не требуется. Если хотите явный v1.2-паритет:

1. Добавьте `sensor_class: SCT_013` в блок `rbamp:` (опционально — это дефолт).
2. Если на UI3 у вас разные клипсы на разных каналах — замените `ct_model:` на `ct_models: [...]`.

Пример UI3 со смешанными клипсами:

```yaml
rbamp:
  id: home_meter
  sensor_class: SCT_013   # дефолт; можно опустить
  ct_models: [SCT_013_005, SCT_013_030, SCT_013_100]
```

Подробнее — в [03_sensor_selection.md](03_sensor_selection.md) и [06_examples.md](06_examples.md) Пример 2.

## v0.3 — 2026-05-28, 8-PR resilience audit

Аудит resilience компонента: сверка с 8 историческими формами багов в экосистеме ESPHome (PR'ы / issues из `esphome/esphome` и `emporia-vue-local/esphome`). Результат: 3 PASS, 3 N/A, 2 WOULD-BREAK (1 исправлен, 1 hardened с проактивным guard'ом).

### Новые тесты

- **`tests/smoke.yaml`** — compile-only smoke-тест для YAML-схемы. Гоняет `MULTI_CONF` с двумя инстансами, все одиночные сенсорные ключи разом, phased-ключи на отдельном инстансе и все поведенческие флаги (`ct_model`, `bidirectional`, `broadcast_latch`, `topology`). Запуск: `esphome compile smoke.yaml` из директории `tests/`. Защищает от silent-регрессий класса `esphome#7081`, где рефакторинг ломает схему без отлова через bench-сессию.

### Hardening

- Длинные flash-write окна (~700 мс на запись модели, ~300 мс пост-ресет) теперь обкармливают watchdog через `App.feed_wdt()`. Защита от будущих PR'ов в планировщик ESPHome, которые могут урезать дефолтный 15-секундный budget `setup()`. Добавлен `#include "esphome/core/application.h"`.
- В example YAML (`ui1`, `ui3`, `multi_module`, `address_change`) добавлены inline-комментарии про безопасные I²C-пины на разных вариантах ESP32 (classic / S2 / S3 / C3). Сам C++ variant-agnostic; рискованной точкой были YAML-дефолты под classic-ESP32.

### Audit verdicts

| Класс | Verdict | Заметка |
|---|---|---|
| Multi-instance state desync (esphome#14152) | PASS | NVS-слот key XOR'ит I²C-адрес + константу версии |
| ADC saturation (esphome#2283) | N/A | Оцифровка на стороне модуля — нет пути через ADC ESP32 |
| Multi-ADC contention (esphome#1915) | N/A | Mutex шины I²C сериализует между инстансами |
| IDF / Arduino SPI mode (esphome#14824) | N/A | Только I²C; нет framework-условного кода |
| Silent schema regression (esphome#7081) | WOULD-BREAK → fixed | Добавлен `tests/smoke.yaml` |
| Per-channel cal cross-talk (esphome#12180) | PASS | Slot-mapping проверен |
| I²C pin defaults across ESP32 variants (emporia-vue#317) | WOULD-BREAK → hardened | YAML-заголовки задокументировали variant-safe пины |
| Boot crash + 0ms watchdog (emporia-vue#411 / #413) | PASS | + проактивный `App.feed_wdt()` guard |

### Breaking changes (v0.3)

Нет. Все изменения additive (новый тестовый файл, новые комментарии, оборонительные WDT-feed'ы без поведенческого влияния в steady state).

### Миграция с v0.2 на v0.3

Действий не требуется. Вызовы `App.feed_wdt()` невидимы для пользователей, не триггерящих `ct_model:` или `new_address:` потоки. Variant-pin комментарии — только документация.

### Credit

Паттерн multi-instance NVS-key derivation (XOR I²C-адреса + версии layout) — самая сильная линия защиты против класса `esphome#14152` (multi-instance cal-pref collision) — был уже в коде до этого аудита. Кредит автору компонента: его паттерн совпал с тем, который позже принял фикс ATM90E32 в #14152 по той же причине (per-instance isolation + stale-data rejection при bump'е layout). Аудит проверил его — не написал.

## v0.2 — 2026-05-24, SPEC §B.5 NACK discipline

Релиз закрывает Phase 1 bench-приёмку и приводит компонент в соответствие с реальностью прошивки v1.0. Готов к production deployment'у для single-phase rbAmp.

### Новые YAML-ключи (v0.2)

- **`topology:`** — `SINGLE` | `SPLIT_PHASE` | `THREE_PHASE`, дефолт `SINGLE`. Информационная подсказка топологии — модуль на текущей прошивке не имеет in-band регистра числа каналов. Идёт в строку `dump_config`. Станет авторитативным когда модуль начнёт публиковать топологию через свой регистр. Дефолт `SINGLE` совпадает со всеми текущими SKU — действий по существующим deployment'ам не требуется.

### Новое поведение

- **NACK retry**: каждый байт float32 регистра ретраится до 3 раз с паузой 5 мс между попытками. Снижает эффективную частоту плохих чтений с ~20% (на 100 кГц из-за NACK-поведения в стеке мастера) до < 1%. См. [SPEC §B.5](https://rbamp.com/docs/modules-basic-standard-api-reference).
- **Слабый sanity-фильтр**: заменяет ранний per-register hard-bound switch. Новый фильтр отбрасывает только non-finite float'ы и значения с `|val| > 10000`. **Нижней границы нет** — brownout, отключение mains (U → 0 В) и off-grid состояния проходят в HA без фильтрации.
- **Warning при `broadcast_latch: true`** при загрузке, когда обнаруженная версия прошивки `< 0x02`. Объясняет, что v1 прошивка тихо сбрасывает записи на широковещательный адрес; компонент откатывается на последовательный latch для каждого модуля. Warning исчезнет автоматически когда прошивка добавит поддержку.

### Bug fixes

- **ESPHome 2025.x API break — `get_object_id_hash` удалён**: предыдущая логика NVS-key seed'а вызывала `this->get_object_id_hash()`, метод убран из `PollingComponent` в 2025.x. Заменено на pattern с XOR'ом I²C-адреса и константы версии layout — даёт per-instance изоляцию и stale-data rejection при bump'е layout. Данные NVS из pre-2025.x deployment'ов инвалидируются этой сменой ключа; Wh-суммы сбрасываются в 0 при первой загрузке после апгрейда.

### Doc updates

- `__init__.py` — блок `CT_MODELS` получил многострочную docstring явно указывающую что `ct_model:` на v1 прошивке — только запись пометки модели.
- `README.md` — секции про запись CT-модели, ограничения broadcast LATCH на v1, использование topology hint и кросс-ссылки на SPEC §B.5 обновлены под реальность v1.
- Example YAML — `i2c: frequency:` поменян с `100kHz` на `50kHz`. Inline-комментарий в каждом файле объясняет почему.
- `example/.gitignore` — добавлен `/bench-*.yaml` чтобы operator-local bench-конфиги с WiFi-кредами и OTA-target'ами не попадали в коммиты.

### Bench-приёмка (2026-05-24)

Пять критериев приёмки проверены на UI1 bench оператора:

| # | Критерий | Результат |
|---|---|---|
| 1 | `esphome compile example/ui1.yaml` clean | PASS (58,71 с, 0 errors) |
| 2 | `dump_config` в пределах 5 с от boot'а | PASS (setup 67 мс; dump_config @ 121 мс) |
| 3 | Мгновенные сущности в пределах ±2% baseline | PASS (U ≈ 228 В, I ≈ 0,9 А, P ≈ 117 Вт, PF ≈ 0,57, F = 50 Гц) |
| 4 | NVS энергия монотонно растёт, переживает перезагрузку | PASS (0,051 → 0,134 → 0,310 → ... → 4,069 Wh через несколько перезагрузок) |
| 5 | Schema validation проходит для всех 4 example YAML | PASS |

Post-fix верификация (6 циклов × 60 с, filter-relaxation run): U = 227-228 В, I = 0,81-0,83 А, P = 100-106 Вт, PF = 0,54-0,56, F = 50 Гц. Ноль `Failed to read` или `i2c.master: NACK` в логе за 6-минутный capture.

### Breaking changes (v0.2)

Нет в плане YAML-конфигурации. Существующие конфигурации `rbamp:` без нового ключа `topology:` продолжают работать (дефолт `SINGLE`). Warning `broadcast_latch:` информационный — изменений конфига не требует.

**NVS key change**: пользователи, апгрейдящиеся с pre-2025.x ESPHome (где использовался старый NVS-путь через `get_object_id_hash`), увидят сброс Wh-сумм в 0 при первой загрузке. Это одноразовое событие. История энергии в long-term statistics HA сохраняется.

## v0.1 — 2026-05-23, baseline import

Baseline-импорт уже существующего ESPHome-компонента в монорепо рядом с кросс-платформенным семейством клиентских библиотек. Этот commit фиксирует компонент в том виде, в котором он существовал до Phase 1 bring-up сессии.

### Что было в baseline

- **YAML-схема `rbamp:`** с ключами `id`, `address`, `update_interval`, `drdy_pin`, `ct_model`, `bidirectional`, `new_address`, `broadcast_latch`. Ключ `topology:` ещё отсутствовал.
- **Сенсорная платформа** с полным набором single-phase и phased полей, mutex-валидатор и валидатор companion-полей.
- **C++ слой компонента** с I²C read helpers, variant detection, NVS persistence, address change, CT model write, latch-pipeline, RT publish. Per-byte NACK retry ещё отсутствовал; sanity-фильтр был простой `isfinite` без `|val| < 10000` cap.
- **Example YAML** (`ui1.yaml`, `ui3.yaml`, `multi_module.yaml`, `address_change.yaml`) с `i2c: frequency: 100kHz`.
- **README.md** — справочник по конфигурации, гид по миграции и секция known-limitations.
- **tests/README.md** — чеклист Phase 1 milestone smoke-тестов.

### Состояние на v0.1

Компонент был функционально завершён для single-phase deployment'ов, но имел три расхождения с реальностью прошивки v1.0, исправленные в v0.2:

1. Не было warning'а при `broadcast_latch: true` (v1 прошивка тихо сбрасывает general-call записи).
2. Docstring `CT_MODELS` не указывала, что v1 `ct_model:` — только metadata, не калибрует NF / gain.
3. Не было ключа `topology:`, не было явного пути topology-подсказки.

Дополнительно, чтение float'ов не имело retry — уязвимо к ~20% частоте NACK на 100 кГц.

### Breaking changes v0.1 → v0.2

См. note про NVS key change в записи v0.2 выше (одноразовый сброс Wh-сумм).

## Forward-совместимость с будущими ревизиями прошивки

Компонент полностью функционален с текущей прошивкой. Когда выйдут планируемые улучшения прошивки, три точки в компоненте получат минорные обновления.

### Точка 1 — определение топологии

**Сейчас**: компонент читает YAML-ключ `topology:` (дефолт `SINGLE`) и использует его как авторитативный источник числа фаз.

**После апгрейда**: компонент будет предпочитать значение топологии, опубликованное модулем через его регистр. Если модуль на старой прошивке вернёт 0 — компонент откатится на YAML-подсказку. Кросс-совместимость v1.0 / v1.2 + v1.x-with-topology-register / v1.x-with-topology-register сохранится.

### Точка 2 — broadcast LATCH

**Сейчас**: при `broadcast_latch: true` и обнаруженной версии прошивки `< 0x02` компонент логирует warning о том, что широковещательные записи сбрасываются модулем, и откатывается на последовательный latch для каждого модуля.

**После апгрейда**: изменений кода не требуется — условие gate'а `firmware_version_ < 0x02` уже обработает это. Когда прошивка поднимет версию выше 0x02, warning перестанет печататься, и широковещательные записи будут проходить штатно.

Проверка: после прошивки модулей новой версией в boot-логе не должно появляться warning'а про "general-call DISABLED", и несколько модулей должны делать latch за одну транзакцию шины.

### Точка 3 — `ct_model:` метаданные → auto-load коэффициентов

**Сейчас v0.4.0 + прошивка v1.2+**: запись `ct_model:` автоматически подгружает noise floor и gain из заводской preset-таблицы на стороне модуля. Никаких calibration-шагов не требуется.

**На ранних прошивках**: запись `ct_model:` — только пометка модели. Compatibility-путь обеспечен через `sensor_class:` precondition: модуль на ранней прошивке примет запись (без auto-load), модуль на v1.2+ обеспечит auto-load автоматически.

### Non-blocking timeline

Ни одно из этих изменений не требуется заранее. Компонент полностью функционален с текущей прошивкой через YAML-подсказку + warning + metadata-only пути. Точки маленькие (< 10 LOC каждая) и могут быть закрыты одним коммитом, когда новая прошивка станет доступна для bench-верификации.

## Гид по апгрейду

### С v0.3 на v0.4.0

1. **Обновите исходники компонента**: `git pull` (если клонировали монорепо) или перепишите директорию `components/rbamp/` целиком.
2. **Опционально — добавьте новые ключи в YAML** для явного v1.2-паритета:

    ```yaml
    rbamp:
      sensor_class: SCT_013         # дефолт; можно опустить
      ct_model: SCT_013_030          # как раньше
      # или для UI3 со смешанными клипсами:
      # ct_models: [SCT_013_005, SCT_013_030, SCT_013_100]
    ```

3. **Никаких breaking-changes**. Старые YAML работают без правок. Действительная разница появится только при работе с модулем на прошивке v1.2+ (auto-load коэффициентов).

### С v0.1 на v0.2 (для исторической справки)

1. **Обновите исходники компонента** (git pull / перепись директории).
2. **Обновите YAML**: `i2c: frequency:` с `100kHz` на `50kHz` во всех конфигах, использующих rbamp.
3. **Прочих изменений YAML не нужно**. Новый ключ `topology:` дефолтится в `SINGLE`, что совпадает со всеми текущими SKU.
4. **NVS Wh обнуление**: первая загрузка после апгрейда с pre-2025.x ESPHome + старой версии компонента покажет `No saved energy state — starting from 0 Wh`. Одноразовое событие из-за смены NVS-ключа. История long-term statistics HA сохраняется.
5. **Уберите `ct_model:` из YAML** если добавили в предыдущей сессии и flash-write уже применён. На v0.3- ключ повторно вызывает flash-write на каждом boot'е (на v0.4.0 + v1.2+ это менее проблемно, но всё равно ~700 мс blackout). На v0.4.0 идемпотентность гарантируется: повторная запись той же модели не делает второй flash-write.

### Из pre-v0.1 (компоненты скопированные до монорепо-импорта)

Pre-монорепо версия компонента не имела NACK retry, NVS-сохранения энергии или enum `CT_MODELS`. Пройдите шаги v0.1 → v0.2 → v0.3 → v0.4.0 выше последовательно; NVS-ключ не пересечётся (namespace ключа выводится из имени, не из per-installation строки).
