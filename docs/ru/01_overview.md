# 01 · Обзор

## Что такое rbAmp

**rbAmp** — компактный аппаратный модуль для прецизионного измерения параметров переменной сети. С точки зрения интегратора это обычный I²C-slave: подайте питание, прочитайте регистр — получите готовое значение в физических единицах. Никакой обработки сигнала на стороне мастера не требуется. Заводская калибровка выполнена один раз на производстве и хранится во flash модуля; пользователю остаётся выбрать модель CT-клипсы (см. [03_sensor_selection.md](03_sensor_selection.md)) и подключить шину.

### Поток данных (в одну строку)

```text
Сеть 230 В → CT-клипса + VT → модуль rbAmp (ADC + DSP) → I²C (физические единицы)
                                                                  ↓
                       Home Assistant ← ESPHome (publish_state) ← компонент rbamp
```

Ключевое УТП: **физические величины приходят с модуля готовыми**. На стороне ESP32 нет ни ADC, ни DSP, ни таблиц калибровки, ни математики RMS/мощности. Компонент ESPHome — это тонкий I²C-клиент + интегратор Wh + слой публикации в HA. Никакой `lambda:`-математики для базовой работы не требуется.

### rbAmp в экосистеме ESPHome

Если вы уже использовали встроенные счётчики энергии (`pzemac` на UART, `atm90e32` на SPI, `cse7766` на UART) — здесь та же самая структура. Компонент `rbamp` — "good citizen" ESPHome: использует стандартный `i2c:` транспорт, стандартный синтаксис `sensor.platform: rbamp`, стандартные сенсорные слоты с `device_class` / `state_class` / `unit_of_measurement`, и поддерживает весь стандартный набор фильтров и свойств сенсоров.

Сравните анатомию (одинаковая ментальная модель):

```yaml
# ATM90E32 (SPI-измеритель):
spi:
  clk_pin: GPIO18
  miso_pin: GPIO19
  mosi_pin: GPIO23

sensor:
  - platform: atm90e32
    cs_pin: GPIO5
    voltage_a:
      name: "Mains Voltage"
    power_a:
      name: "Mains Power"

# rbAmp (I²C-измеритель):
i2c:
  sda: GPIO21
  scl: GPIO22
  frequency: 50kHz

rbamp:
  id: meter1
  address: 0x50

sensor:
  - platform: rbamp
    rbamp_id: meter1
    voltage:
      name: "Mains Voltage"
    power:
      name: "Mains Power"
```

Различия — только в транспорте (`i2c:` вместо `spi:`) и в том, что `rbamp:` объявляется отдельным блоком (нужен `id:` + `address:` для multi-module шины). Дальше — обычный ESPHome: имена слотов (`voltage`, `current`, `power`, `energy`, `frequency`, `power_factor`) узнаваемые, кастомизация (`name`, `update_interval`, `accuracy_decimals`, `filters:`) работает как обычно. Подробный пример кастомизации — в [06_examples.md §1.1 "UI1 с кастомизацией — все стандартные ESPHome-крутилки"](06_examples.md).

## Что вы измеряете

| Величина | YAML-слот | Диапазон | Типичная точность |
|---|---|---|---|
| Среднеквадратичное напряжение | `voltage` | 0..300 В | ±0,5% |
| Среднеквадратичный ток (на канал) | `current` / `current_1` / `current_2` | 0..100 А (зависит от SKU) | ±0,5..1% |
| Активная мощность (со знаком) | `power` / `power_1` / `power_2` | ±23 кВт | ±1% |
| Коэффициент мощности (со знаком) | `power_factor` / `power_factor_1` / `power_factor_2` | −1..+1 | ±0,01 |
| Реактивная мощность | `reactive_power` / `reactive_power_1` / `reactive_power_2` | как у P | ±2% |
| Полная мощность | `apparent_power` (только CH0; для CH1/CH2 — template-сенсор HA, см. [07_diy_integrations.md](07_diy_integrations.md) §6) | как у P | ±2% |
| Частота сети | `frequency` | 45..65 Гц | ±0,5 Гц |
| Накопленная энергия (Wh) | `energy` / `energy_1` / `energy_2` | 64-bit | зависит от точности P |

Все мгновенные величины обновляются модулем автономно с шагом ~200 мс. Накопление Wh ведёт ESP32 на основе средней мощности, возвращаемой модулем за интервал между двумя `update()`. ESPHome-компонент скрывает протокол latch/snapshot — пользователь видит только готовые сенсорные сущности в Home Assistant.

## Что добавляет компонент ESPHome

- **Готовые HA-сенсоры** — у каждого слота уже выставлены `device_class`, `state_class` и единицы. Сенсор `energy` сразу появляется в Energy-дашборде без ручной конфигурации.
- **NVS-сохранение Wh** — суммарная энергия пишется во flash ESP32 каждые 5 минут и восстанавливается **до** первого `publish_state`. Home Assistant никогда не видит секундного нуля после перезагрузки и не интерпретирует его как сброс счётчика.
- **Декларативная конфигурация датчика** — две YAML-строки (`sensor_class:` + `ct_models:` для UI3 или `ct_model:` для UI1) фиксируют семейство и модель CT для каждого канала. Заводские коэффициенты подгружаются модулем автоматически — никаких calibration-шагов в загрузочной последовательности.
- **Расписание опроса** через `PollingComponent::update()` с настраиваемым `update_interval` (по умолчанию 60 с).
- **Устойчивость к NACK** — реализована трёхслойная дисциплина (SPEC §B.5): 50 кГц по умолчанию, per-byte retry, sanity-фильтр на NaN/Inf. Подробности — ниже.
- **Однократная провизия адреса** — ключ `new_address:` меняет I²C-адрес модуля за один boot-цикл. Удобно для bulk-настройки партии устройств.
- **Multi-module шина** — `MULTI_CONF = True` позволяет объявить несколько блоков `rbamp:` на одной ESP32 и опрашивать их все по очереди.

## Архитектура

```
  230 В сеть
       |
  +----+----+
  | CT-клипса |  (SCT-013 или встроенный шунт — зависит от SKU)
  | трансформатор напряжения |  (только на UI* SKU)
  +---------+
       |   гальваническая развязка
  +----|----+
  |   ADC + DSP    дискретизация и вычисление RMS/P/PF
  |  + I²C-slave   адрес 0x50 (по умолчанию), 50/100 кГц
  +---------+
       |   I²C (SDA/SCL + опционально DRDY)
  +----|----+
  |  ESP32  |
  |  ESPHome firmware (этот компонент)
  |  PollingComponent::update() каждые 60 с
  |  master-side Wh-накопление
  +---------+
       |   Wi-Fi (ESPHome native API, port 6053)
  +----|----+
  |  Home Assistant
  |  Energy-дашборд, автоматизации, история
  +---------+
```

Измерительный engine живёт в модуле; ESP32 выступает мостом к HA и интегратором энергии. Wh считаются на ESP32, не в модуле — это важно для multi-module установок: часы ведущего стабильнее, чем низкочастотный осциллятор модуля, и накопление по всем модулям остаётся консистентным.

## Поток данных за один `update()`

Каждый вызов `update()` выполняет две параллельные ветки:

### Ветка 1 — мгновенные значения (RT block)

1. Проверка status-регистра. Если модуль занят (boot, flash-write blackout) — пропуск без записи.
2. Для каждого объявленного RT-сенсора — чтение float32 из соответствующего регистра.
3. Sanity-фильтр `std::isfinite()` — отброс NaN/Inf, которые могут возникнуть из-за tearing при чтении не-атомарных регистров.
4. `sensor->publish_state(val)`.

Шина модуля не поддерживает auto-increment, поэтому компонент читает каждый из 4 байт float32 отдельной транзакцией с собственным address-phase. Это согласуется с конвенцией SPEC §6, которой следуют все клиентские библиотеки семейства.

### Ветка 2 — period metering (накопление энергии)

1. Зафиксировать `t_latch = millis()`.
2. Отправить latch-команду — закрывает текущий период накопления и сразу открывает следующий. Одна транзакция закрывает и открывает; на границе ни одного отсчёта не теряется.
3. Через 50 мс (non-blocking timeout) — прочитать valid-флаг. Если модуль не успел проинтегрировать (опрос быстрее периода интегрирования или stale snapshot) — пропуск, следующий цикл подберёт пропущенный интервал.
4. Прочитать среднюю активную мощность за период по каждому каналу.
5. `E_Wh[ch] += avg_p_W[ch] × (t_latch − last_latch_ms_) / 3_600_000`.
6. Обновить `last_latch_ms_` только после успеха по всем каналам.
7. Сохранить в NVS, если прошло > 5 минут с последнего save.

Первый latch после boot сбрасывает накопитель модуля и калибрует часы мастера; его результат отбрасывается через флаг `primed_`.

## NVS-сохранение энергии

Накопленные значения хранятся в структуре `RbAmpPrefData` (per-channel total + per-channel export):

```cpp
struct RbAmpPrefData {
  double energy_total_wh[3];
  double energy_export_wh[3];
} __attribute__((packed));
```

NVS-ключ выводится из ID компонента, XOR'енного с `RBAMP_PREF_VERSION` (`0x52424D31` = "RBM1"). Если структура когда-нибудь поменяется — bump константы версии тихо отбросит устаревшие данные вместо их неправильной интерпретации.

При загрузке `setup()` восстанавливает суммы **до** первого `update()`. Если бы компонент сначала опубликовал `0 Wh`, а потом прыгнул к сохранённому значению, Home Assistant интерпретировал бы прыжок как сброс счётчика и начал новый сегмент истории. Восстановление до первого publish это предотвращает.

Частота сохранения определяется двумя условиями:

- **По времени** — каждые 5 минут (`SAVE_INTERVAL_MS = 300000`).
- **По событию** — после каждого успешного полного цикла интегрирования (все каналы прочитаны без NACK).

Худший случай потери данных при внезапном пропадании питания — 5 минут накопленной энергии. При средней нагрузке 60 Вт это 5 Wh — ниже разрешения дневного бара Energy-дашборда.

### NVS wear — может ли flash «износиться»?

Каждое сохранение пишет ~16 байт в NVS-секцию ESP32. ESP-IDF NVS layer
группирует мелкие записи в один блок и стирает страницу (4 КБ) только
когда блок заполнен. На 5-минутном цикле:

| Метрика | Значение |
|---|---|
| Сохранений в сутки | 288 |
| Сохранений в год | ~105 120 |
| Размер записи | ~16 байт |
| Размер NVS-страницы | 4 096 байт |
| Page-erase в год | ~411 (≈ 105 120 × 16 ÷ 4 096) |
| Лимит цикла стирания (datasheet ESP32 NOR-flash) | 100 000 циклов |
| Time-to-fail (TTF) | ~240 лет на одну NVS-страницу |

ESP-IDF NVS дополнительно делает wear-leveling по доступным NVS-страницам
(дефолт раздел — 4 страницы), что умножает TTF на ~4×. Практический
вывод: износ NVS на 5-минутном save-цикле не лимитирующий фактор —
аппаратура устареет раньше.

**Когда стоит уменьшить частоту**: если пользователь форсирует
`SAVE_INTERVAL_MS` ниже дефолта (например 30 с) через свой
кастомный код — пересчитайте TTF по той же формуле. Безопасный нижний
порог для 10-летней эксплуатации — ~30 с.

## Дисциплина SPEC §B.5

Платы ESP32 на ESP-IDF v5 драйвере `i2c_master` (на котором работают ESPHome, arduino-esp32 v3.x и MicroPython на ESP32) на скорости 100 кГц периодически NACK'ают чтения с модуля — примерно в 20% транзакций. Тот же самый DUT, тот же модуль, но на STM32 F103 или Raspberry Pi в качестве мастера — 0% NACK. То есть проблема на стороне мастера, не модуля. Подробности — в SPEC Appendix B.5.

Есть второе осложнение: когда `i2c_master_transmit_receive()` возвращает `ESP_FAIL` из-за NACK, драйвер IDF v5 **не** обнуляет пользовательский буфер. Остаются данные предыдущей транзакции. Некоторые из этих stale-паттернов выглядят как правдоподобные float'ы (например `0x3C 0x2F 0xFB 0x3F` → 1.962 В) и проходят наивную проверку `isfinite()`.

Компонент реализует три слоя митигации:

| Слой | Реализация | Эффект |
|---|---|---|
| 50 кГц шины по умолчанию | `frequency: 50kHz` в YAML-блоке `i2c:` | Уменьшает NACK-rate ~в 5 раз эмпирически |
| Per-byte retry | 3 попытки, пауза 5 мс, на каждый байт каждого float32 | Снимает transient NACK до использования данных |
| Слабый sanity-фильтр | `std::isfinite()` без жёстких физических границ | Ловит NaN/Inf/денормали из stale-буфера; пропускает корректные edge-case значения (brownout, disconnect) |

С тремя активными слоями компонент достигает > 99% валидных чтений на bench. На 100 кГц текущий NACK-rate всё ещё высок для надёжной работы; когда в прошивке выйдет фикс stop-handler hardening, частоту можно будет вернуть на 100 кГц одной строкой в YAML.

## Семейство библиотек rbAmp

Этот компонент — один из членов кросс-платформенного семейства клиентских библиотек rbAmp. Все они реализуют один и тот же wire-протокол ([`libs/spec/SPEC.md`](https://rbamp.com/docs/modules-basic-standard-api-reference)).

| Платформа | Путь в монорепо | Состояние |
|---|---|---|
| **ESPHome** (этот компонент) | `tools/esphome-rbamp/` | Стабильно, HA-сенсоры проверены |
| **Arduino / arduino-esp32** | `libs/arduino/RbAmp/` | Стабильно |
| **ESP-IDF** | `libs/esp_idf/components/rbamp/` | Стабильно, v1.2-паритет |
| **Python (CPython + MicroPython)** | `libs/python/rbamp/` | Стабильно, v1.2-паритет |
| **STM32 HAL** | `libs/stm32_hal/` | В разработке |

> Пути показывают расположение в монорепо. Distribution-репозитории для отдельной установки через Library Manager / pip / `idf_component_manager` готовятся к публикации — соответствующие URL появятся в [11_changelog.md](11_changelog.md) после релиза.

Все библиотеки используют одинаковую конвенцию single-byte-per-transaction чтения (SPEC §6), один и тот же стейт-машин period-metering (SPEC §7), один и тот же алгоритм определения варианта SKU (SPEC §8), одни и те же коды ошибок (SPEC §5). Код с любой из этих платформ переносится на другую механической трансляцией синтаксиса — протокольная логика идентична.

## Поддерживаемые SKU

| Вариант | Канал напряжения | Каналов тока | Мощность | Доступные ESPHome-слоты |
|---|---|---|---|---|
| UI1 | да | 1 | да | `voltage`, `current`, `power`, `energy`, `frequency`, `power_factor`, `reactive_power`, `apparent_power` |
| UI2 | да | 2 | да | + `current_1`, `power_1`, `energy_1`, `power_factor_1`, `reactive_power_1` |
| UI3 | да | 3 | да | + `current_2`, `power_2`, `energy_2`, `power_factor_2`, `reactive_power_2` |
| I1 | нет | 1 | нет | `current`, `frequency` |
| I2 | нет | 2 | нет | + `current_1` |
| I3 | нет | 3 | нет | + `current_2` |

На I-only вариантах слоты `power`, `energy` и `power_factor` требуют объявленного слота `voltage` (проверяется валидатором схемы) — объявляйте их только на UI* SKU.
