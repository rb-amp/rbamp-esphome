# 04 · Подключение

Эта глава о физическом подключении rbAmp к ESP32: выбор GPIO, правила pull-up'ов, многомодульная шина, питание, опциональный DRDY и провизия адресов.

## Распиновка LV-стороны

Низковольтная сторона модуля выводит пять сигналов. Все гальванически развязаны от мейнсов — пин `GND` это GND ESP32, не нейтраль сети.

| Пин | Сигнал | Описание |
|---|---|---|
| `VCC` | +4,5..+5,5 В DC | Питание модуля. 5 В номинал; ниже 4,5 В возможна потеря точности ADC. |
| `GND` | Общий | Общий с мастером. Обязателен. |
| `SDA` | I²C data | Логика 3,3 В; 5 В-толерантен. |
| `SCL` | I²C clock | Логика 3,3 В; 5 В-толерантен. |
| `DRDY` | Готовность данных (опционально) | Open-drain выход. LOW-импульс ~10 мкс каждые ~200 мс после обновления мгновенных регистров. |

## Выбор GPIO на ESP32

Подойдёт любая пара GPIO, поддерживающих I²C. Матрица периферии ESP32 позволяет назначить I²C почти на любой пин. Практические ограничения:

- **GPIO0** и **GPIO2** — boot-strapping пины. Низкий уровень на них при включении блокирует загрузку. Не использовать.
- **GPIO 6..11** — на большинстве ESP32-WROOM подключены к встроенной flash или PSRAM. Недоступны.
- **GPIO 34..39** — input-only. Не подходят для I²C (нужен двусторонний драйв).
- Не подключайте ESP32 GPIO к каким-либо reset-линиям модуля — сброс модуля делается I²C-командой, никаких аппаратных reset-выводов на стороне мастера не требуется.

Рекомендованные дефолтные назначения, работающие на большинстве ESP32-DevKitC:

| Сигнал | GPIO | Заметка |
|---|---|---|
| SDA | GPIO21 | Дефолт во всех example YAML |
| SCL | GPIO22 | Дефолт во всех example YAML |
| DRDY | GPIO15 | Опционально; подойдёт любой input-capable пин |

Если GPIO21/22 заняты — выберите любую другую валидную пару и обновите блок `i2c:` в YAML.

## Подключение одного модуля

```
  ESP32                        rbAmp
  ------                       +-----------+
  GPIO22 (SCL) ────────────── SCL          |
  GPIO21 (SDA) ────────────── SDA          |
  5V           ────────────── VCC          |
  GND          ────────────── GND          |
                               |           |
  (DRDY опционально)           |           |
  GPIO15 ◄──[10K к 3.3V]───── DRDY         |
                               +-----------+
```

Внешние pull-up резисторы для одного модуля не нужны — на плате rbAmp уже стоят встроенные 4,7 кОм на SDA и SCL к 3,3 В.

## Многомодульная шина

Когда на одной шине два и более модуля (или другие I²C-устройства со своими pull-up'ами), их подтяжки складываются параллельно. Суммарное сопротивление падает ниже эффективного минимума — что приводит к:

- Излишнему потреблению по шине.
- Возможной перегрузке open-drain выходов ESP32 (типично ≤ 3 мА при 3,3 В).
- Повышенной ёмкостной нагрузке на длинной шине.

На нижней стороне каждой PCB rbAmp есть solder-jumper'ы, помеченные `Pull-Up`. Перерезка дорожки острым лезвием отключает встроенную пару 4,7 кОм на этом модуле.

| Модулей на шине | Конфигурация pull-up'ов | Эффективное R |
| --- | --- | --- |
| 1 | Встроенные оставить. Внешние не нужны. | ~4,7 кОм |
| 2 | Оставить только на первом, у второго перерезать. | ~4,7 кОм |
| 3 | Перерезать у всех. Внешняя пара **2,7..3,3 кОм** у ESP32. | 2,7..3,3 кОм |
| 4..6 | Перерезать у всех. Внешняя пара **2,2..2,7 кОм** у ESP32. | 2,2..2,7 кОм |
| 7..16 | Перерезать у всех. Внешняя пара **1,5..2,2 кОм** у ESP32 + проверка ёмкости шины (см. ниже). | 1,5..2,2 кОм |
| ESP32 уже даёт pull-up'ы | Перерезать у всех модулей. Внутренние pull-up'ы ESP32 (≥50 кОм) слабые — на длинной шине всё равно нужны внешние 2,2..4,7 кОм. | как указано выше |

**Логика выбора**: эффективное R_pullup — параллельное соединение всех активных подтяжек. Целевой диапазон 1,5..4,7 кОм (минимум ограничен sink-током ESP32 ≤ 3 мА на 3,3 В; максимум — RC-постоянной шины). Формула: 1/R_eff = N/R_unit. Для встроенной 4,7-кОм пары: 2 модуля = 2,35 кОм, 3 = 1,57 кОм, 4 = 1,18 кОм — на 3 модулях уже становится низковато, поэтому таблица обнуляет встроенные jumpers начиная с 3 модулей и переводит на единственную внешнюю пару.

**Диагностическая проверка**: измерьте сопротивление между SDA и VCC при выключенном питании. Результат должен быть в диапазоне **1,5..4,7 кОм**. Меньше 1,0 кОм — слишком много активных подтяжек параллельно; больше 10 кОм — нет активной подтяжки (проверьте перемычки).

**Ёмкость шины** (для ≥7 модулей или общей длины проводки > 50 см): I²C-spec ограничивает суммарную C_b ≤ 400 пФ. Короткая шина с 2-3 модулями — типично 50..100 пФ; шина 1 м с 10 модулями может подняться до 300..400 пФ. При близких к лимиту значениях падайте на `frequency: 50kHz` (если ещё на 100 кГц) или используйте active-pullup IC (например PCA9517).

## Скорость шины

Установите `frequency: 50kHz` в блоке `i2c:` ESPHome. На текущей прошивке это **обязательно** для надёжной работы с ESP-IDF v5:

```yaml
i2c:
  sda: GPIO21
  scl: GPIO22
  frequency: 50kHz    # обязательно — не используйте 100 кГц на ESP32 с v1 прошивкой
```

Модуль периодически NACK'ает чтения с ESP32 на 100 кГц (~20% транзакций) из-за взаимодействия таймингов драйвера ESP-IDF v5 `i2c_master` с post-STOP recovery-окном I²C-периферии модуля (детали — в [`libs/spec/SPEC.md` Приложение B.5](https://rbamp.com/docs/modules-basic-standard-api-reference)). На 50 кГц частота NACK падает ~в 5 раз, и per-byte retry внутри компонента вытягивает > 99% валидных чтений.

Не повышайте до 100 кГц на ESP32 пока в прошивке не выйдет фикс stop-handler hardening — после этого ограничение можно будет снять одной строкой в YAML.

Другие I²C-мастера (STM32 HAL, Raspberry Pi smbus2, Arduino UNO / Nano) этой проблемой не страдают и могут работать на 100 кГц.

## Эталонная схема — один ESP32 + один rbAmp

```
                                         rbAmp
                                     +-------------------+
  ESP32                              |   LV-сторона      | Mains-сторона
  -----                              |                   |
  GPIO22 (SCL) ──────────────────── SCL                  |◄[клемма L]── L
  GPIO21 (SDA) ──────────────────── SDA                  |◄[клемма N]── N
  GPIO15 (input) ◄──[10K к 3.3V]── DRDY (open-drain)     |
  3.3V ─────────────────────────── (опора подтяжки выше) |◄[CT-jack]── SCT-013
  5V ──────────────────────────── VCC                    |
  GND ─────────────────────────── GND                    |
                                     +-------------------+
```

Заметки:

- Резистор 10 кОм на DRDY стоит на стороне мастера (около GPIO), не на стороне модуля. DRDY — open-drain, внутренней подтяжки нет.
- CT-jack должен быть вставлен полностью; клипса полностью охватывает один проводник стрелкой по направлению к нагрузке.
- Полярность L/N на клеммах влияет на знак P (см. [03_sensor_selection.md](03_sensor_selection.md)).

## Питание

### Питание модуля (VCC)

Модуль требует +4,5..+5,5 В DC. Потребление по току:

| Режим | Типичное |
|---|---|
| Установившееся измерение | ~15 мА |
| Запись во flash (при `ct_model:` / провизии адреса) | ~25 мА пик, ~700 мс |
| Boot warm-up (~250 мс) | ~15 мА |

Пин 5V на ESP32-DevKitC (обычно подключённый к USB VBUS) может питать модуль напрямую — USB-хост должен тянуть суммарный ток ESP32 (~200 мА) + rbAmp (~15 мА пик) = ~215 мА. Для инсталляций где ESP32 питается от внешнего 5 В SMPS — подключайте VCC к той же 5 В шине.

### Brownout ESP32

Brownout-детектор ESP32 срабатывает на ~2,43 В на VDD. При срабатывании I²C-шина переходит в idle, а модуль продолжает измерять автономно (у него собственное 5 В питание). Когда ESP32 восстанавливается — `setup()` восстанавливает накопленную энергию из NVS до первого `update()`. Home Assistant не видит ни разрыва, ни сброса в графике энергии.

## Опциональный DRDY

Пин DRDY на модуле — open-drain. Он даёт LOW-импульс ~10 мкс каждые ~200 мс сразу после обновления блока мгновенных регистров. На period-snapshot не срабатывает.

При `update_interval: 60s` DRDY можно опустить — компонент проверяет статус-регистр в начале каждого `update()` и пропускает чтение если модуль не готов. DRDY полезен только когда нужна sub-секундная доставка мгновенных данных с минимальной латентностью (interrupt-driven чтения).

### Разводка DRDY

DRDY — open-drain: тянет LOW, но не может драйвить HIGH. Нужен внешний pull-up:

```
  rbAmp DRDY  ──────────────────┬── ESP32 GPIO (input)
                                 |
                               [10K]
                                 |
                               3.3V
```

10 кОм к 3,3 В. **Не подключайте DRDY к 5 В**, даже если SDA/SCL 5 В-толерантны — DRDY это отдельный сигнал, и абсолютный максимум на input GPIO ESP32 равен 3,6 В.

Объявите пин в YAML:

```yaml
rbamp:
  id: meter1
  drdy_pin: GPIO15
```

### Multi-module DRDY

Сигналы DRDY от нескольких модулей можно wire-OR'ить на один GPIO ESP32 (open-drain выходы естественно объединяются). Общий сигнал идёт в LOW когда **любой** модуль обновил мгновенные регистры. Компонент использует это только как подсказку и не различает кто именно сработал.

Если нужны раздельные сигналы — заведите DRDY каждого модуля на отдельный GPIO и объявите `drdy_pin:` в каждом блоке `rbamp:` отдельно.

## Провизия адресов на многомодульной шине

Все модули rbAmp поставляются с завода с адресом `0x50`. До подключения нескольких модулей на одну шину каждому нужно выдать уникальный адрес.

Последовательная провизия:

1. Подключите только модуль 1 к ESP32 (один на шине).
2. Залейте конфигурацию из [`example/address_change.yaml`](../../example/address_change.yaml) с `address: 0x50` и `new_address: 0x51`. При загрузке компонент обнаружит несоответствие и выполнит последовательность смены адреса: запись нового адреса, сохранение во flash (~700 мс), мягкий ресет (~300 мс), верификация на новом адресе.
3. Подтвердите успех в логах ESPHome. Обновите YAML: `address: 0x51`, уберите `new_address:`.
4. Отключите модуль 1. Подключите модуль 2 (всё ещё на заводском `0x50`).
5. Повторите для `0x52`, `0x53` и т.д.
6. Когда у всех модулей уникальные адреса — подключите их все на шину. Перережьте pull-up перемычки на всех кроме одного (см. правила выше).

Последовательная провизия избегает коллизий адресов в процессе смены. Запись нового адреса требует **factory-provisioning mode** на стороне модуля — см. соответствующий раздел ниже.

## Пример YAML для multi-module

```yaml
i2c:
  sda: GPIO21
  scl: GPIO22
  frequency: 50kHz
  scan: true

rbamp:
  - id: meter_house
    address: 0x50
    update_interval: 60s
  - id: meter_solar
    address: 0x51
    update_interval: 60s
    bidirectional: true
  - id: meter_evcharger
    address: 0x52
    update_interval: 60s

sensor:
  - platform: rbamp
    rbamp_id: meter_house
    voltage: { name: "House Voltage" }
    current: { name: "House Current" }
    power:   { name: "House Power" }
    energy:  { name: "House Energy" }
  - platform: rbamp
    rbamp_id: meter_solar
    voltage: { name: "Solar Voltage", internal: true }
    current: { name: "Solar Current" }
    power:   { name: "Solar Power" }
    energy:  { name: "Solar Energy" }
    energy_exported: { name: "Solar Energy Exported" }
  - platform: rbamp
    rbamp_id: meter_evcharger
    voltage: { name: "EV Charger Voltage", internal: true }
    current: { name: "EV Charger Current" }
    power:   { name: "EV Charger Power" }
    energy:  { name: "EV Charger Energy" }
```

Полная версия с аннотациями + `broadcast_latch:` forward-compat — в [`example/multi_module.yaml`](../../example/multi_module.yaml).

## Длина шины

I²C — это короткобазовый протокол. Проверенные длины для rbAmp:

| Тип кабеля | Максимальная длина | Скорость |
|---|---|---|
| Стандартный JST / 4-жильный flat | 0,3 м | 50 кГц |
| Витая пара (SDA+GND и SCL+GND в разных парах) | 1 м | 50 кГц |
| Витая пара + I²C-буфер (PCA9515 / TCA9617) | 3 м | 50 кГц |
| Дифференциальная шина (PCA9615 / LTC4332) | 100 м | 50 кГц |

Для длиннее 0,3 м используйте витую пару, причём SDA и SCL должны лежать в **разных** парах (например синий + бело-синий для SDA, зелёный + бело-зелёный для SCL). Если SDA и SCL в одной паре — кросс-связь искажает фронты уже на 50 кГц.

## Factory-provisioning mode

Некоторые операции модуля (смена I²C-адреса, factory reset) требуют, чтобы модуль был в **factory-provisioning mode** — специальном режиме для одноразовых провизионных действий. Стандартные production-модули поставляются с **отключённым** provisioning-режимом: попытка смены адреса на них вернёт ошибку, видимую в логах ESPHome как `WARNING` при загрузке.

Если вам нужна смена адреса в полевых условиях, обратитесь к документации модуля или поставщику за процедурой временного включения provisioning-режима. После завершения провизии модуль возвращается в обычный production-режим.

> Конфигурация датчика тока (`ct_model:` / `ct_models:`) в factory-provisioning mode **не** требует — на прошивке v1.2+ модуль принимает запись CT-модели напрямую (см. [03_sensor_selection.md](03_sensor_selection.md)).
